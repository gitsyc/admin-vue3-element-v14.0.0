# SRS需求规格说明书生成器

## Agent 协作架构

```
req-doc skill（主流程：调度 + 文件写入）
  ├── req-analyzer agent（分析师）
  │     职责：读取设计文档/代码/已有需求，输出结构化功能摘要
  │     输出：字段表、业务规则、枚举值、依赖关系
  │
  ├── req-writer agent（撰写者）
  │     职责：摘要 + 规范 → 符合 PRD 语言的 Markdown 章节
  │     约束：只写"做什么"，不写技术实现，不写 UI 布局
  │
  └── req-reviewer agent（审查者）
        职责：四维度独立审查（语言规范/章节结构/三要素/提示文案）
        → 有 ❌ 项 → 反馈给 req-writer 修正 → 重新审查（最多2次）
        → 全部通过 → 主流程写入文件
```

**多功能并行**：若需要生成多个功能模块，Step A4 可并行启动多个 `req-analyzer`，每个负责一个功能，分析完成后串行调用 `req-writer` + `req-reviewer`（写作和审查需要串行保证质量）。

---

## 工作模式

| 模式 | 触发场景 | 入口 |
|------|---------|------|
| **生成模式** | 从零生成SRS需求规格说明书 | → Step A |
| **反向同步模式** | 根据已有前端代码更新SRS需求规格说明书 | → Step B |
| **局部完善模式** | 补充或修改某个章节 | → Step C |
| **审查修复模式** | 审查已有SRS需求规格说明书，找出问题并修复 | → Step D |

---

## Step A: 生成模式

### A1: 扫描项目上下文

先主动扫描项目，找到已有文档再开始工作：

| 优先级 | 文件类型 | 查找方式 |
|--------|---------|---------|
| 最高 | SRS需求规格说明书 | `Glob("**/*需求*说明书*.md")` |
| 高 | 设计方案 | `Glob("**/*设计*方案*.md")` |
| 低 | 路由/代码 | `src/router/`, `src/views/`, `src/api/` |

### A2: 读取模板

读取 `references/templates/requirements-spec.md`，严格按照模板中每个章节的内容要求生成文档。

### A3: 收集信息

**必须收集：** 项目名称、项目背景和目标、核心功能模块列表、目标用户群体

**按需询问：** 系统类型（Web/移动端/多端）、特殊业务规则

### A4: 生成任务清单

**必须先展示任务清单，再逐个生成，禁止一次性写入整个文档。**

任务拆分原则：**每个功能模块（3.5.x）是一个独立任务，非详细设计章节（文档信息、项目概述、功能清单等）也各自独立。**

任务清单示例：
```
| 序号 | 任务名称 | 类型 | 状态 |
|------|---------|------|------|
| 1  | 文档信息 + 项目概述 | [文档] | [ ] 等待中 |
| 2  | 生成业务流程图 | [图表] | [ ] 等待中 |
| 3  | 3.1 总体功能架构 | [文档] | [ ] 等待中 |
| 4  | 生成系统架构图 | [图表] | [ ] 等待中 |
| 5  | 3.2 需求功能清单 | [文档] | [ ] 等待中 |
| 6  | 3.3 页面功能清单 | [文档] | [ ] 等待中 |
| 7  | 3.4 页面访问权限 | [文档] | [ ] 等待中 |
| 8  | 生成工作台操作流程图 + 时序图 | [图表] | [ ] 等待中 |
| 9  | 3.5.1 工作台 | [Agent] | [ ] 等待中 |
| 10 | 生成数字大屏操作流程图 + 时序图 | [图表] | [ ] 等待中 |
| 11 | 3.5.2 数字大屏 | [Agent] | [ ] 等待中 |
| 12 | 生成设备档案操作流程图 + 时序图 | [图表] | [ ] 等待中 |
| 13 | 3.5.3.1 设备档案 | [Agent] | [ ] 等待中 |
...
```

**关键规则：每个 [Agent] 任务之前必须有对应的 [图表] 任务，[图表] 任务完成后将图片路径记录在备注列，[Agent] 任务执行时从备注列读取路径传给 req-writer。**

状态标识：[ ] 等待中 → [进行中] → [完成]

**类型说明：**
- [文档]：主流程直接写（文档信息、概述、功能清单等结构性章节）
- [图表]：调用 `diagram-generator` 技能生成
- [Agent]：调用 analyzer → writer → reviewer 流程生成（所有 3.5.x 功能详细设计）

### A5: 执行任务

**执行规则：**
1. 每次只执行一个任务，执行前更新为 [进行中]，完成后更新为 [完成]
2. 第一个任务用 Write 创建文件，后续用 Edit 追加
3. 每次写入不超过 150 行

**[文档] 类任务执行方式（主流程直接写）：**

直接按模板写入，无需调用 agent。

**[图表] 类任务执行方式：**

调用 `diagram-generator` 技能生成对应图表，获取图片路径后记录在任务清单备注中，供后续 [Agent] 任务使用。

**[Agent] 类任务执行方式（3.5.x 功能详细设计）：**

每个功能模块按以下流程执行：

```
Step 1: 调用 req-analyzer
  subagent_type: "req-analyzer"
  传入：
    PROJECT_PATH: {项目根目录}
    FEATURE_NAME: {功能名称}
    MODE: from_design（有设计文档）或 from_code（无设计文档）
    SOURCE_PATH: {设计文档路径 或 src/ 目录}

Step 2: 调用 req-writer
  subagent_type: "req-writer"
  传入：
    PROJECT_PATH: {项目根目录}
    FEATURE_NAME: {功能名称}
    SECTION_NUMBER: {章节编号，如 3.5.3.1}
    功能摘要: {req-analyzer 的完整输出}
    DIAGRAM_PATHS: {对应 [图表] 任务已生成的图片路径，若未生成则传空}

Step 3: 调用 req-reviewer
  subagent_type: "req-reviewer"
  传入：
    PROJECT_PATH: {项目根目录}
    FEATURE_NAME: {功能名称}
    章节内容: {req-writer 的完整输出}

Step 4: 处理审查结果
  - 若无 ❌ 项 → 直接进入 Step 5
  - 若有 ❌ 项 → 将审查报告传给 req-writer 修正，重新审查（最多2次）
    超过2次仍有问题 → 主流程自行修正后写入，并在任务清单备注问题

Step 5: 写入文件
  用 Edit 工具将 req-writer 的输出追加到文档
```

**注意：** diagram-generator 是技能（Skill），不是 agent，无法在 agent 协作流程中间调用。流程图必须作为独立的 [图表] 类任务，在对应 [Agent] 任务之前完成，再将图片路径传给 req-writer。

### A6: 质量检查（全文档生成后）

所有任务完成后，主流程执行最终检查：

1. 检查 3.3 与 3.5 的对应性（每个 3.3 条目在 3.5 中都有章节）
2. 检查文档结构完整性（对照模板目录）
3. 发现问题立即用 Edit 修复

---

## Step B: 反向同步模式

**触发场景：** 前端功能已实现，需要将实际功能同步回SRS需求规格说明书。

### B1: 定位目标

确认要同步的范围（指定模块或全量扫描）。

### B2: 并行分析

对每个需要同步的功能模块，并行调用 `req-analyzer`：

```
subagent_type: "req-analyzer"
传入：
  PROJECT_PATH: {项目根目录}
  FEATURE_NAME: {功能名称}
  MODE: from_code
  SOURCE_PATH: {src/ 目录}
```

### B3: 对比差异

收到所有 analyzer 的摘要后，与已有SRS需求规格说明书对比：

| 差异类型 | 处理方式 |
|---------|---------|
| 需求有、代码无 | 标注说明，询问用户是否删除 |
| 代码有、需求无 | 调用 req-writer 补充到SRS需求规格说明书 |
| 字段不一致 | 以代码为准，调用 req-writer 更新 |
| 业务规则缺失 | 调用 req-writer 补充 |

### B4: 更新文档

展示差异清单，用户确认后：
- 对需要更新的章节，走 req-writer → req-reviewer → 写入 流程
- 更新历史版本表格

---

## Step C: 局部完善模式

1. 读取现有SRS需求规格说明书对应章节
2. 调用 `req-analyzer`（MODE: from_existing）分析现有内容，识别缺失和问题
3. 调用 `req-writer` 补充/改写
4. 调用 `req-reviewer` 审查
5. 通过后用 Edit 工具更新文件

---

## Step D: 审查修复模式

### D1: 扫描文档

找到SRS需求规格说明书：`Glob("**/*需求*说明书*.md")`

### D2: 生成审查任务清单

**先展示审查任务清单，再逐章审查，禁止一次性完成所有审查。**

审查任务清单示例：
```
| 序号 | 审查项 | 类型 | 状态 |
|------|-------|------|------|
| 1  | 一、文档信息 | [主流程] | [ ] 等待中 |
| 2  | 二、项目概述（2.1-2.5） | [主流程] | [ ] 等待中 |
| 3  | 三、3.1-3.4 结构性章节 | [主流程] | [ ] 等待中 |
| 4  | 三、3.5.1 工作台 | [Agent] | [ ] 等待中 |
| 5  | 三、3.5.2 数字大屏 | [Agent] | [ ] 等待中 |
| 6  | 三、3.5.3.1 设备档案 | [Agent] | [ ] 等待中 |
...
```

### D3: 逐章审查

- **结构性章节**（文档信息、概述、功能清单）：主流程直接对照模板检查
- **3.5.x 功能详细设计**：调用 `req-reviewer` 独立审查

```
subagent_type: "req-reviewer"
传入：
  PROJECT_PATH: {项目根目录}
  FEATURE_NAME: {功能名称}
  章节内容: {从文档中读取的对应章节内容}
```

### D4: 汇总审查报告并自动进入修复

所有章节审查完后，汇总输出审查报告，**然后立即自动进入 D5，无需询问用户是否修复**。

```
## 审查报告

### P0 结构性问题（必须修复）
- [ ] 问题描述 → 修复方案

### P1 内容缺失（需要补充）
- [ ] 问题描述 → 修复方案

### P2 规范不符（建议优化）
- [ ] 问题描述 → 修复方案
```

> 审查的目的就是修复。输出报告后直接生成修复任务清单并开始执行，不等待用户确认。
> 唯一例外：若用户在调用时明确说"只审查，不修复"，则在报告后停止。

### D5: 生成修复任务清单并立即执行

将所有 ❌ 问题按章节归组，生成修复任务清单，**展示后立即开始执行第一个任务**：

```
| 序号 | 修复项 | 问题数 | 修复方式 | 状态 |
|------|-------|-------|---------|------|
| 1  | 3.5.3.1 设备档案 | 3个问题 | [Agent 重写] | [ ] 等待中 |
| 2  | 3.5.3.3 厂商信息 | 5个问题 | [Agent 重写] | [ ] 等待中 |
| 3  | 3.3 页面功能清单 | 1个问题 | [主流程修正] | [ ] 等待中 |
```

**修复方式判断：**
- [Agent 重写]：问题数 >= 2，或涉及语言规范/章节结构/三要素缺失 → 整章重写
- [主流程修正]：问题数 = 1，且是局部文案/格式问题 → 直接 Edit 修改

**关键：D3 审查时每章的问题清单必须保留在上下文中，D6 修复时直接传给 req-writer 和 req-reviewer，不需要重新审查。**

### D6: 逐项执行修复

每次只修复一个任务，执行前更新为 [进行中]，完成后更新为 [完成]，立即开始下一个。

**[Agent 重写] 流程（整章重写）：**

```
Step 1: 调用 req-analyzer（MODE: from_existing）
  传入：
    PROJECT_PATH: {项目根目录}
    FEATURE_NAME: {功能名称}
    MODE: from_existing
    SOURCE_PATH: {SRS需求规格说明书路径}
  作用：提取现有章节中可复用的内容，同时标注哪些需要改写

Step 2: 调用 req-writer
  传入：
    PROJECT_PATH: {项目根目录}
    FEATURE_NAME: {功能名称}
    SECTION_NUMBER: {章节编号}
    功能摘要: {req-analyzer 的输出}
    审查报告: {D3 中该章节的问题清单，原文传入，不要省略}
    DIAGRAM_PATHS: {已有流程图路径，无需重新生成}

Step 3: 调用 req-reviewer 复审
  传入：
    PROJECT_PATH: {项目根目录}
    FEATURE_NAME: {功能名称}
    章节内容: {req-writer 的输出}
    原始问题清单: {D3 中该章节的 ❌ 问题列表，原文传入}

Step 4: 处理复审结果
  - 全部通过 → 进入 Step 5
  - 仍有 ❌ → 再次调用 req-writer 修正（最多重试1次）
    仍未通过 → 写入文件并在文档末尾附注"待人工复查：[问题描述]"

Step 5: 替换文档中的对应章节
  用 Read 读取文档，定位该章节的起止行号
  用 Edit 将原章节内容替换为 req-writer 的新输出
  替换后验证：Read 文档对应位置，确认替换成功
```

**[主流程修正] 流程（局部修改）：**

```
直接用 Edit 工具修改对应行
修改后 Read 验证内容正确
```

### D7: 修复完成后更新文档版本

所有修复任务完成后：
1. 更新文档历史版本表格（版本号 +0.1，说明"审查修复：修正 N 处规范问题"）
2. 汇总输出修复结果：

```
## 修复完成

共修复 N 个章节，M 个问题：
✅ 3.5.3.1 设备档案 — 修复3个问题（语言规范2个、提示文案1个）
✅ 3.5.3.3 厂商信息 — 修复5个问题（章节结构缺失3个、三要素2个）
✅ 3.3 页面功能清单 — 修复1个问题（格式）

文档版本已更新至 V{x.x}
```
