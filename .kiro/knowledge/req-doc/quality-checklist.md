# SRS需求规格说明书质量审查清单

> 供 req-reviewer agent 使用。逐项检查，每项必须明确标注 ✅ 通过 / ❌ 不通过（含具体问题位置）。

---

## 维度一：语言规范（每句话逐一检查）

### 1.1 禁止词汇扫描

搜索文档中是否包含以下词汇，发现即为不通过：

**技术实现类：**
- [ ] API、接口、mock、axios、request
- [ ] 路由、router、组件、component、props、emit
- [ ] a-、el-、van-（UI 组件前缀）
- [ ] flex、grid、margin、padding、px、%、rgba、#（CSS）
- [ ] 数据库、SQL、字段名（英文）、JSON

**UI 组件类：**
- [ ] Modal、Drawer、抽屉、弹窗（应替换为"弹出编辑窗口"等）
- [ ] Tag、标签组件、Steps、步骤条、Tab、页签
- [ ] TreeSelect、树形下拉、Radio、Checkbox、Switch、开关组件
- [ ] tooltip、hover（应替换为"悬停显示"）

**数据来源类：**
- [ ] 数据源：/xxx（接口路径）
- [ ] 前端过滤、不请求接口
- [ ] 调用接口、API 请求、数据库查询

### 1.2 业务术语一致性

- [ ] 同一概念在全文使用统一术语（启用/停用不与开启/关闭混用等）
- [ ] 数据来源描述用业务语言（"来源于[模块]中已启用的[数据]"）

---

## 维度二：章节结构完整性

### 2.1 对照模板检查每个 3.5.x 功能模块

每个功能模块必须包含：

- [ ] **（一）功能需求概述**：存在且不含技术词汇
- [ ] **筛选功能**：有字段表格（筛选字段 | 类型 | 说明）
- [ ] **列表功能**：有字段表格（列名 | 字段类型 | 说明），含分页说明和空数据提示文案
- [ ] **新增功能**：有字段表格（字段名称 | 类型 | 长度限制 | 必填 | 默认值 | 说明），含提交成功/失败提示文案
- [ ] **编辑功能**：有说明（字段同新增，或列出差异），含提交成功提示文案
- [ ] **删除功能**：有确认文案原文，有关联保护说明（如有）
- [ ] **业务规则**：每条规则包含触发条件 + 影响范围 + 提示文案原文
- [ ] **边界与异常**：包含空数据场景、权限不足场景
- [ ] **（三）业务逻辑关联**：说明跨模块依赖关系
- [ ] **（四）（五）流程图**：有图片引用占位

### 2.2 3.3 与 3.5 对应性

- [ ] 3.3 页面功能清单中列出的每个页面，在 3.5 中都有详细设计
- [ ] 3.5 中的每个页面，在 3.3 中都有对应条目
- [ ] 3.3 中列出的子功能（筛选/列表/新增/编辑/删除等），在 3.5 中都有对应描述

---

## 维度三：功能描述三要素

每个功能描述必须包含（逐功能检查）：

- [ ] **数据定义**：展示/操作什么数据，数据来源和含义
- [ ] **业务规则**：什么条件下显示/隐藏，数据如何计算/判断
- [ ] **交互功能**：用户能做什么操作，操作后发生什么

---

## 维度四：提示文案完整性

- [ ] 每个新增/编辑操作有成功提示文案（原文）
- [ ] 每个删除操作有确认文案（原文）和成功提示文案（原文）
- [ ] 每个有唯一性校验的字段有错误提示文案（原文，含占位符如【XXX】）
- [ ] 每个有关联保护的删除有阻止提示文案（原文，含数量占位符如 X）
- [ ] 每个列表有空数据提示文案（原文）
- [ ] 每个有权限控制的功能说明了隐藏方式（隐藏/禁用）

---
