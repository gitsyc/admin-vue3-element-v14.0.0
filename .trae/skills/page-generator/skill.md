# 页面生成器工作流

每次生成功能页面，必须严格按以下 6 个步骤执行，每步都要有结果输出。

> **重要**：本 skill 是通用工作流框架，不绑定任何特定项目或框架。
> 项目特有约定（目录结构、命名规范、路由方式、导航方式、全局样式等）
> 全部从 `knowledge/project-conventions.md` 读取。
> 若该文件不存在，则在步骤 2 中通过读取项目文件自行推断。

> **多子项目说明**：若工作目录下存在多个子项目（如 `admin/`、`mobile/`），
> 步骤 1 必须先确定目标子项目，后续所有路径均以**子项目根目录**为基准。

---

## 步骤 1：需求理解

**目标**：明确要实现什么，不能靠猜。

执行动作：
1. **探测子项目结构**：检查当前工作目录下是否存在多个含 `package.json` 的子目录（如 `admin/`、`mobile/`、`web/` 等）
   - 若存在多个子项目，根据用户描述的功能判断目标子项目；若无法判断，询问用户
   - 确定后，将该子项目目录作为后续所有步骤的 **PROJECT_PATH**（例如 `/path/to/project/admin`）
   - 若只有一个子项目或根目录本身就是项目，直接用根目录作为 PROJECT_PATH

2. 检查项目中是否有需求说明书（常见位置：`docs/`、根目录 `.md` 文件）
2. 如果有，**按以下顺序读取对应章节**（需求说明书遵循 req-doc 模板结构）：

   **第一步：读取菜单结构**（`3.1 总体功能架构` 的菜单结构文字说明）
   - 确认目标功能所属的一级菜单、二级菜单
   - 确认路由路径（从菜单层级推断，不能自己猜）

   **第二步：读取页面功能清单**（`3.3 页面功能清单`）
   - 找到目标功能对应的所有行
   - 提取该功能下的所有子功能（筛选、列表、新增、编辑、删除、导出、状态切换、特殊操作等）
   - 这是功能的完整清单，后续实现必须覆盖所有子功能，不能遗漏

   **第三步：读取功能详细设计**（`3.5 功能详细设计` 对应章节）
   - 逐字读取，提取每个子功能的完整设计：
     - 筛选字段（字段名、控件类型、数据源）
     - 列表字段（列名、字段类型、渲染方式）
     - 表单字段（字段名、控件类型、是否必填、长度限制、默认值、数据源）
     - 特殊交互（动态增删行、联动显隐、级联、文件上传等）
     - 操作反馈（成功/失败提示文案）
     - 业务规则（删除限制、状态联动、唯一性校验、关联关系等）
     - 边界与异常（空数据提示、权限控制、错误提示）
     - 业务逻辑关联（依赖哪些其他模块的数据）

   **第四步：读取数据字典**（`6.1 数据字典`，如有）
   - 提取相关枚举值定义（状态值、类型值等）

3. 如果没有需求文档，根据用户描述整理以上信息，并主动询问不明确的功能点

**输出格式**：
```
📖 需求理解
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
功能：[功能名称]
菜单层级：[一级菜单] > [二级菜单]
路由路径：[路径]
页面布局：[布局类型]

功能点清单：
  ✅ [功能点1，如：列表展示，含字段：xxx | xxx | xxx]
  ✅ [功能点2，如：按名称/状态筛选]
  ✅ [功能点3，如：新增/编辑弹窗，表单字段：xxx（必填）、xxx（非必填）]
  ✅ [功能点4，如：删除，有关联保护]
  ✅ [功能点5，如：状态切换，停用后影响登录]
  ✅ [功能点N，如：重置密码，确认弹窗，成功提示xxx]
  ...（需求里有多少写多少，不要遗漏）

数据依赖：
  - [依赖的外部数据，如：角色列表（/api/role/list，已存在）]
  - [依赖的外部数据，如：设备分类树（/api/device-category/tree，需新建）]

业务规则：
  - [规则1]
  - [规则2]

边界与异常：
  - [异常1]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️ 如有不确定的功能点，在此列出并等待用户确认后再继续
```

## 步骤 2：规范加载与组件规划

**目标**：调用 `spec-loader` agent 按需加载规范并规划组件，只读取当前功能实际需要的内容。

执行动作：
使用 `Agent` 工具，`subagent_type: "page-spec-loader"`，传入以下 prompt（将占位符替换为实际值）：

```
项目路径：{当前项目根目录绝对路径}

功能需求：
{步骤 1 输出的完整需求理解内容}
```

> `page-spec-loader` agent 定义在 `.claude/agents/page-spec-loader.md`，
> 它会自动检测 UI 库、按需加载对应规范、规划组件，返回结构化摘要。

**输出格式**：
```
📚 规范加载与组件规划
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[spec-loader agent 返回的规范摘要，原样展示]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 步骤 3：设计决策

**目标**：在写代码前确定视觉风格，避免生成"AI 默认丑页面"。

执行动作：
使用 `Agent` 工具，`subagent_type: "design-advisor"`，传入：

```
PAGE_TYPE: {admin / mobile / dashboard / landing，从步骤 1 和步骤 2 推断}
COMPONENT_LIB: {组件库名称和版本，来自步骤 2}
FEATURE_NAME: {功能名称}
PROJECT_PATH: {步骤 1 确定的子项目根目录绝对路径}
```

> `design-advisor` 会调用 `ui-ux-pro-max` 数据库查询适合当前页面类型的风格和 UX 规则，输出结构化设计决策。

**输出格式**：
```
🎨 设计决策
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[design-advisor agent 返回的设计决策，原样展示]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

> 步骤 5 生成代码时，必须将以上设计决策作为约束，不得忽略。

---

## 步骤 4：执行计划

**目标**：列出所有需要创建/修改的文件，确认无遗漏。

执行动作：
1. 根据需求和规范，确定需要创建/修改的文件清单
2. 判断路由归属（属于哪个模块，是否需要新建路由文件）
3. 判断导航是否需要更新（侧边栏/tabbar/顶部导航/路由配置等，取决于项目类型）
4. 判断是否需要新建目录

**输出格式**：
```
📋 执行计划
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
功能：[功能名称]

任务清单：
  ⬜ 1. [mock 文件路径]          Mock 数据
  ⬜ 2. [api 文件路径]           API 接口定义
  ⬜ 3. [views 文件路径]         页面组件
  ⬜ 4. [router 文件路径]        路由（如需新建模块）
  ⬜ 5. [router/index 路径]      注册路由
  ⬜ 6. [导航文件路径]           更新导航菜单
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 步骤 5：任务清单生成

**目标**：用任务清单方式逐文件生成，主线程直接写，不用 Agent（避免重复加载上下文的开销）。

执行规则：
- 先展示完整任务清单，每个任务开始前更新状态为 🔄，完成后更新为 ✅
- 按顺序逐文件生成，每个文件写完后立即标记完成
- **所有具体路径、命名规范、代码风格，严格按步骤 2 加载的规范执行**

**任务清单格式**：
```
📋 生成任务清单
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⬜ 1. mock/xxx.js              Mock 数据（持久化）
⬜ 2. src/api/xxx.ts           API 接口定义
⬜ 3. src/views/Xxx/index.vue  页面组件
⬜ 4. src/router/modules/xxx.ts 路由模块（如需新建）
⬜ 5. src/router/index.ts      注册路由
⬜ 6. AppSidebar/index.vue     更新导航菜单
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**生成要求**：
- Mock 数据：模块级变量持久化，初始数据用真实业务数据，增删改查全部实现
- API 文件：每个导出函数前写 JSDoc 注释
- 页面组件：
  - 严格遵循步骤 2 加载的**组件库规范**（Ant Design Vue / Vant / Element Plus 等）
  - 同时遵循步骤 2.5 的**设计决策**（间距、圆角、UX 规则、移动端专项）
  - 移动端页面必须：字体 ≥ 14px、触摸目标 ≥ 44px、有空状态、有加载状态、有底部安全区
  - 只加有实际功能的按钮，不加多余 UI 库自定义属性
- 多功能时（如设备管理含3个子功能），按功能分组依次完成，每组内顺序：mock → api → views

---

## 步骤 6：路由与导航

**目标**：注册路由、更新导航，确保功能可访问。

执行顺序（必须串行，有依赖关系）：

**5.1 路由注册**
- 按项目规范（从 project-conventions.md 或步骤 2 推断）注册路由
- 可能是：新建路由模块文件 → 在入口文件注册；或直接在单一路由文件追加；或文件路由（无需手动注册）

**5.2 导航更新**
- 按项目类型更新对应导航：
  - **Web 管理后台**：更新侧边栏菜单文件，按需求文档菜单结构确定归属，若分组不存在则新建
  - **移动端 H5**：更新底部 tabbar 配置（如有），或仅通过路由跳转访问（无需更新 tabbar）
  - **PC 前台网站**：更新顶部导航配置（如有）
  - **无固定导航**：跳过此步骤，通过路由直接访问
- 图标/图片资源按项目规范选择

**输出格式**：
```
🔧 后处理
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 路由：[操作描述]
✅ 导航：[操作描述，或"无需更新"]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

---

## 步骤 7：验收检查

**目标**：调用 `page-reviewer` agent 从四个维度独立审查，确保交付质量。

执行动作：
使用 `Agent` 工具，`subagent_type: "page-reviewer"`，传入以下 prompt：

```
项目路径：{当前项目根目录绝对路径}

功能需求：
{步骤 1 输出的完整需求理解内容}

规范摘要：
{步骤 2 spec-loader 返回的规范摘要}

生成的文件列表：
{步骤 5 任务清单中所有已完成的文件路径}

开发服务器端口：{从 vite.config.ts 读取，或默认 3000}
```

> `page-reviewer` agent 定义在 `.claude/agents/page-reviewer.md`，
> 它会独立检查代码报错、需求完整性、Mock 接口、规范合规，返回审查报告。

**收到审查报告后**：
- 若有 ❌ 项，主流程立即修复对应文件，修复后重新调用 `page-reviewer` 复查该项
- 直到所有项目全部 ✅ 才算完成

## 参考资源索引

> **规范库位置**：`.claude/knowledge/`（项目级共享，所有 skill 和 agent 均可使用）
> **加载方式**：Step 2 通过 `spec-loader` agent 按需加载，不会一次性读取所有文件。

| 规范文件 | 用途 | 加载时机 |
|---------|------|---------|
| `knowledge/conventions/project.md` | 项目目录结构、命名规范、全局样式、导航结构 | 始终加载 |
| `knowledge/conventions/mock.md` | Mock 数据持久化写法、CRUD 模板 | 始终加载 |
| `knowledge/ui-libs/ant-design-vue/components.md` | Ant Design Vue 组件用法规范 | 仅 ant-design-vue 项目 |
| `knowledge/ui-libs/ant-design-vue/pages.md` | Ant Design Vue 页面布局规范 | 仅 ant-design-vue 项目 |
| `knowledge/ui-libs/element-plus/components.md` | Element Plus 组件用法规范 | 仅 element-plus 项目 |
| `knowledge/ui-libs/element-plus/pages.md` | Element Plus 页面布局规范 | 仅 element-plus 项目 |
| `knowledge/ui-libs/vant/components.md` | Vant 组件用法规范 | 仅 vant 项目 |
| `knowledge/ui-libs/vant/pages.md` | Vant 页面布局规范 | 仅 vant 项目 |

> 新增 UI 库只需在 `knowledge/ui-libs/` 下添加对应目录，`spec-loader` 会自动发现。
> 新增规范类型（业务组件、API 规范等）放在 `knowledge/` 对应子目录，按需在 agent 中引用。
