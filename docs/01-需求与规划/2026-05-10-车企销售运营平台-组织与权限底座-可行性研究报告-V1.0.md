## 一、现状及必要性

### 1.1 项目背景

车企销售运营平台面向总部、区域、经销商与门店多层级组织协同运行，业务覆盖线索到成交再到回款、交付与车主运营。由于参与主体多、组织层级深、权限边界复杂，若缺少统一的组织与权限底座，将导致跨层级协作效率低、数据口径不一致、关键操作缺少留痕等问题，进而影响总部经营监控与精细化运营。主要痛点体现在：一是组织边界不清，总部/区域/经销商/门店之间的数据隔离与下钻范围缺少统一规则，造成“看得到/看不到”的争议与风险；二是权限口径不一致，菜单权限、操作权限、数据权限与关键业务权限分散配置，新增组织或人员时配置成本高且容易越权；三是关键操作不可追溯，价格、作废、调拨、退款等高风险业务动作缺少审计留痕将难以满足合规要求与事后追责；四是组织变更影响不可控，门店划转、人员调岗等调整会影响数据归属与报表统计口径，缺少可追溯策略会造成经营数据口径混乱。为解决上述问题，建设“组织与权限底座”模块，统一组织架构、权限模型与数据权限策略，为各业务域提供一致的访问控制与审计能力。

### 1.2 国内外现状及发展趋势

**国内现状：** 国内车企与经销商体系普遍已建设DMS/CRM/线索管理等系统，但在总部-区域-渠道多层级协同中，常见问题是权限与数据口径分散在各业务系统内，组织主数据与角色体系难以复用，权限变更成本高且风险大。随着精细化运营与跨域协同需求提升，统一的组织主数据与权限底座逐步成为平台化建设的基础能力。

**国外现状：** 国外成熟企业在销售运营与渠道管理中更强调统一身份与访问控制体系，通常将组织、用户、角色与审计能力下沉为平台基础服务，并通过标准化API供各业务域复用，以保障合规与跨域协同效率。

**发展趋势：** 组织与权限领域呈现“平台化+精细化”趋势：细粒度授权（RBAC与数据范围策略结合、逐步引入ABAC）、全链路审计与合规可视化、云原生与弹性伸缩、统一数据治理口径与主数据管理、与单点登录/多因素认证/外部身份源集成等能力逐步成为标配。

### 1.5 需求和必要性分析

**业务需求分析：** 车企销售运营平台覆盖“线索→跟进→试驾→报价→订单→收款→交付→车主运营”的全链路业务，涉及总部、区域、经销商与门店多主体协同。组织主数据需要统一维护总部→区域→经销商→门店→员工的层级关系与人员归属；权限需要同时覆盖菜单/操作权限、数据范围（区域/门店/个人等）与关键业务权限（价格、作废、调拨、退款等高风险动作），并要求在权限不足时“禁止访问且隐藏入口”，避免越权与误操作。随着组织调整、人员流动频繁，需支持角色复用、批量授权与复制，降低配置成本，并在门店划转等变更时明确历史数据统计口径（按发生时组织/按当前组织）以保证经营口径一致。

**必要性分析：**（1）**业务发展需要**：平台化建设要求各业务域在同一组织与权限体系下协同运行，才能支撑总部全局视角、区域下钻分析与经销商/门店日常操作的并存；（2）**降本增效需要**：统一底座能力可复用角色与权限策略，减少重复建设与人工配置成本，提升新增组织/人员、调整权限的交付效率；（3）**合规安全需要**：最小权限原则与默认隔离是数据安全底线，关键变更与高风险操作必须留痕可追溯、可检索、可导出，以满足审计与追责；（4）**决策支持需要**：组织与统计口径一致是指标体系与报表下钻的前提，组织变更可追溯能够保障经营数据的连续性和可解释性；（5）**自主研发必要性**：组织权限模型需要深度契合车企多层级渠道管理与业务规则（含组织调整口径策略、关键业务权限项等），通过自主研发可确保可控可扩展、避免对单一供应商产品能力边界的依赖，满足持续迭代与平台化复用需求。

## 二、建设目标及内容

### 2.1 顶层设计及总体规划

车企销售运营平台总体目标是打通“线索 → 跟进 → 试驾 → 报价 → 订单 → 收款 → 交付 → 车主运营”的业务闭环，并实现总部/区域/经销商/门店/员工的协同体系与数据口径统一，支撑总部经营监控与精细化运营。本项目建设内容属于平台基础能力层，定位为“组织与权限中心（底座）”，为线索运营、库存协同、销售中心、财务管理、报表驾驶舱等业务域提供统一的组织主数据、权限模型、数据隔离与审计留痕能力，减少各域重复建设与权限口径割裂风险。

### 2.2 项目建设目标

#### 2.2.1 业务及功能目标

本项目建设完成后，实现以下业务及功能目标：

1. **统一组织架构与人员归属**：支持总部→区域→经销商→门店→员工多级组织维护，明确人员归属与上下级关系，支撑跨层级管理与业务协同。
2. **形成可复用的角色与权限体系**：建立菜单权限、操作权限、数据权限与关键业务权限的统一模型，支持按角色批量授权与快速复制，降低权限配置成本。
3. **保障默认数据隔离与可控放开**：以经销商隔离、门店仅看本店、个人仅看本人为默认原则，并支持按角色配置放开范围，满足总部全局视角与区域下钻需求并存。
4. **实现关键操作全量留痕**：对组织、人员、角色、权限策略及高风险业务权限等关键变更记录操作人、时间、对象与字段变更明细，支持检索与导出。
5. **支持组织调整并明确统计口径**：门店划转、组织合并等变更时，必须选择历史数据统计口径策略（按发生时组织/按当前组织），并形成可追溯记录。

#### 2.2.2 系统性能目标

- 用户规模：支持不少于 5000 名用户账号规模（总部/区域/经销商/门店）。
- 并发用户数：支持不少于 300 个并发访问。
- 响应时间：列表查询 95% 请求在 1 秒内返回（不含网络波动）；普通配置保存 < 3 秒。
- 权限生效：角色或权限调整保存后，对新请求即时生效。
- 系统可用性：建议不低于 99.9%。

#### 2.2.3 系统运维目标

- 运维响应时效：7×24 小时（核心时段优先保障）。
- 故障修复时间：一般故障 < 4 小时，严重故障 < 1 小时（视运维资源配置可调整）。
- 运维范围：覆盖组织、员工、角色、权限策略、审计日志与组织调整等核心配置能力，确保变更可追溯、可回溯、可审计。

### 2.3 项目开发功能

| 序号 | 功能模块 | 功能范围说明 | 优先级 |
|-----|---------|-------------|-------|
| 1 | 组织管理 | 维护总部/区域/经销商/门店多级组织，支持启用/停用与层级关系约束 | P0 |
| 2 | 员工管理 | 员工账号、归属组织、岗位与状态维护，支持调岗与离职处理 | P0 |
| 3 | 角色管理 | 角色维护、授权与复制，支持批量授权与快速复用 | P0 |
| 4 | 权限策略 | 菜单/操作/数据/关键业务权限配置与校验，默认隔离并可控放开 | P0 |
| 5 | 审计日志 | 关键变更留痕查询、字段变更明细下钻与导出 | P0 |
| 6 | 组织调整 | 门店划转/组织合并等调整与口径策略管理，形成可追溯记录 | P1 |

### 2.4 项目实施内容

- 需求梳理与确认：梳理组织层级、角色体系、数据范围策略与关键业务权限项清单，明确口径与边界。
- 方案设计：组织主数据模型、权限模型与校验规则、审计留痕模型、组织调整口径策略设计。
- 系统开发：组织/员工/角色/权限策略/审计/组织调整等功能开发与联调，提供标准化API供业务域复用。
- 数据准备与初始化：组织、人员、角色与权限策略的初始化导入与校验，必要时提供批量导入工具。
- 测试与验收：覆盖权限越权、数据隔离、组织变更口径、审计留痕完整性等关键场景；组织UAT与回归。
- 上线与推广：灰度发布、权限切换策略、上线培训与运行保障。

### 2.5 项目运维方案

- 账号与权限运维：建立角色模板与变更流程，权限变更需审计留痕并支持回溯。
- 日志与审计：审计日志可检索与导出，保留期限按合规要求执行；组织调整记录永久保留。
- 备份与恢复：组织主数据与权限配置按日备份，支持按时间点恢复与差异核对。
- 监控与告警：登录失败、权限校验异常、关键配置变更、接口错误率与响应时间等纳入监控告警。

## 三、技术方案

### 3.1 技术路线

#### 3.1.1 总体架构

建议采用B/S架构与前后端分离模式，组织与权限底座以“基础能力服务+管理端”方式提供：管理端用于组织、人员、角色与策略配置；服务端对外提供统一鉴权、数据范围计算、审计留痕等能力接口，供各业务域复用。

![](images/system-architecture.png)

#### 3.1.2 应用架构

- 管理端应用：PC端管理后台（Web应用），面向总部/区域/经销商管理员等角色进行配置与审计查询。
- 基础服务（建议）：组织服务、权限服务、审计服务，提供REST API并支持与业务域服务调用集成。
- 业务域应用：线索、销售、库存、财务、报表等业务域在统一权限底座约束下运行。

#### 3.1.3 数据架构

- 主数据：组织（Org）、员工（Employee）、角色（Role）、权限项（Permission）、数据范围策略（DataScopePolicy）。
- 过程数据：角色授权记录、权限变更记录、组织调整记录、审计日志（含字段变更明细）。
- 口径数据：组织调整口径策略（按发生时组织/按当前组织）作为报表与历史归属的口径依据。

#### 3.1.4 技术架构

- 前端（现有工程栈）：Vue 3 + TypeScript + Vite + Element Plus，状态管理使用 Pinia，网络请求使用 Axios。
- 后端（建议）：Java/Spring Boot 或 Node.js 服务化架构，对外提供RESTful API；权限校验作为统一中间件/网关能力下沉。
- 数据存储（建议）：关系型数据库（如MySQL）存储组织、账号、角色与策略等配置数据；缓存（如Redis）用于权限结果缓存与热点组织树缓存。

#### 3.1.5 集成架构

- 统一鉴权：各业务域在接口层接入统一的鉴权与数据范围计算能力，避免各域重复实现与口径不一致。
- 账号体系扩展：支持后续与单点登录/外部身份源对接（本期预留扩展点）。
- 审计集成：对组织、人员、角色、策略与关键业务权限变更统一写入审计服务，支持跨域检索与导出。

#### 3.1.6 部署架构

建议采用至少“应用双实例+数据库主备/高可用”的部署形态：管理端静态资源通过Nginx等Web服务器发布；服务端应用支持横向扩展；数据库与缓存部署在可靠环境中并纳入备份与监控。对于关键配置能力，建议通过灰度与回滚机制降低变更风险。

#### 3.1.7 性能指标

- 列表查询响应时间：95% 请求在 1 秒内返回（不含网络波动）。
- 权限生效：角色或权限调整保存后，对新请求即时生效。
- 审计写入：关键变更审计写入成功率不低于 99.9%，失败需具备补偿或告警机制。

#### 3.1.8 自主可控要求

组织与权限底座涉及全平台访问控制与数据隔离的核心能力，建议采用自主可控的权限模型与策略引擎：权限口径统一、策略可配置、可审计、可扩展，并通过标准接口向业务域输出能力，避免业务域“各自为政”带来的安全与合规风险。

### 3.2 数据建设方案

- 数据字典：统一组织类型（总部/区域/经销商/门店）、组织与账号状态、数据范围（全部/区域/经销商/门店/个人）、统计口径策略（按发生时组织/按当前组织）、审计动作类型等枚举口径。
- 数据质量与约束：组织编码、账号、角色编码全局唯一；组织层级关系满足约束（总部仅顶层、门店不可为上级）；停用组织需校验下级组织与在岗账号状态。
- 初始化与迁移：支持组织、人员、角色与策略的批量初始化导入；导入过程提供冲突校验与回滚策略，避免一次性变更引发业务中断。

### 3.3 系统安全建设方案

- 身份认证：账号密码登录；账号停用后禁止登录；连续多次登录失败触发安全策略（如锁定/验证码/告警）。
- 权限控制：基于角色的访问控制，菜单权限控制可访问菜单，操作权限控制可执行操作，数据权限控制可查询与可处理范围，关键业务权限控制高风险操作；无权限访问禁止进入并提示，无权限操作项不展示。
- 数据隔离：经销商之间默认不可互看；门店默认仅看本店；个人默认仅看本人；跨层级下钻以数据权限策略授权范围为准。
- 敏感信息保护：密码与敏感字段不得以明文方式展示给非授权人员；接口建议采用HTTPS并对关键操作进行防重放与风控校验（视整体安全体系落地）。
- 审计留痕：对组织、员工、角色、权限策略、关键业务权限等变更必须记录留痕，包含操作人、操作时间、对象类型、对象标识、动作类型、变更字段明细与结果；审计记录支持检索与导出。

### 3.4 技术标准规范

- 接口规范：统一REST API风格、鉴权方式、错误码与返回结构；关键操作必须具备幂等与审计ID。
- 权限规范：统一权限点命名规则（菜单/操作/关键业务权限分层），数据范围策略统一解释口径并在业务域复用。
- 数据规范：主数据编码规则、枚举字典统一管理，组织调整口径策略作为指标口径治理的一部分纳入评审。
- 研发与交付规范：代码规范、分支与发布规范、变更评审与灰度策略、回滚预案与应急流程。

### 3.5 关键技术

- 组织树与层级约束：支持多级组织的维护与校验，保障组织关系正确性与可扩展性。
- 统一权限模型与生效规则：多个角色下菜单/操作权限合并生效，数据权限按并集生效，关键业务权限按“任一角色具备即具备”生效（可配置）。
- 数据范围计算与下钻：实现区域/经销商/门店/个人等范围的统一计算，支撑总部全局视角与区域/门店下钻并存。
- 审计变更明细：对关键对象变更形成字段级差异记录，满足合规与追责需要。
- 组织调整口径策略：支持组织变更时选择历史数据统计口径（按发生时组织/按当前组织）并可追溯。

## 六、可行性分析

### 6.1 技术论证

- 技术可行：组织与权限底座以配置管理、鉴权校验、数据范围计算与审计留痕为核心，属于成熟的企业级通用能力范畴；前端可基于现有 Vue 3 + TypeScript + Element Plus 技术栈构建管理后台，具备良好的组件生态与开发效率。
- 架构可扩展：通过将组织、权限、审计能力服务化并以标准API输出，可支撑线索、销售、库存、财务等业务域复用，降低重复建设与口径割裂风险。
- 安全与合规可落地：基于最小权限原则与默认隔离策略，结合字段级审计留痕与导出能力，可满足“可追溯、可审计”的合规要求，并为后续SSO/外部身份源对接预留扩展空间。

### 6.2 效益分析

- 管理效益：统一组织主数据与权限口径，减少总部/区域/经销商/门店之间的权限争议，提升跨层级协同效率与管理透明度。
- 业务效益：以底座方式支撑全链路业务闭环，保障总部经营看板与报表下钻的组织口径一致性，提升经营分析与决策效率。
- 合规效益：关键配置与高风险权限变更全量留痕，可检索、可导出，降低越权与审计风险，支持事后追责与复盘。
- 成本效益：角色模板与批量授权降低新增组织/人员、权限调整的配置成本；底座能力复用减少各业务域重复开发与维护成本。

### 6.3 风险评估

| 风险项 | 风险描述 | 影响 | 应对措施 |
|-------|---------|------|---------|
| 需求口径分歧 | 总部、区域、经销商对可见范围/下钻口径理解不一致 | 配置策略反复、影响上线 | 设立口径评审机制，明确默认隔离与可控放开原则；P0范围先冻结 |
| 权限模型复杂 | 菜单/操作/数据/关键业务权限组合复杂，易产生越权或误配 | 安全与合规风险 | 统一权限点规范与校验规则，提供冲突检测与保存校验，强化测试用例 |
| 组织调整影响面大 | 门店划转、合并导致历史数据归属与报表口径变化 | 指标波动与信任风险 | 强制选择口径策略并留痕；上线前完成口径宣贯与模拟校验 |
| 集成落地风险 | 各业务域接入统一鉴权与数据范围计算改造量不一致 | 影响统一口径 | 提供标准SDK/中间件与接入指南；分域分批接入并设立验收门槛 |
| 审计数据规模 | 审计留痕长期保留，数据量增长带来存储与查询压力 | 性能与成本压力 | 分区/归档策略、索引优化、冷热分离与导出异步化（按需纳入） |
